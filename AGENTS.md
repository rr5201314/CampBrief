# CampBrief 项目规范

## 项目定位

CampBrief 是面向大学生的信息聚合静态站，当前阶段以 HTML、CSS、JavaScript 直接构建，优先部署到 GitHub Pages。后续迁移到云服务器时，应保持前端静态资源可直接由 Nginx、Caddy 或其他静态文件服务托管。

## 目录约定

- `index.html` 是站点首页。
- `pages/<module>/index.html` 存放独立栏目页，模块目录使用英文小写 kebab-case。
- `assets/css/` 存放全站样式和页面样式。
- `assets/js/` 存放全站脚本和页面脚本。
- `assets/images/` 存放可公开访问的图片资源。
- `assets/data/` 或 `data/` 用于存放可公开的静态 JSON 数据。
- `docs/` 用于存放项目文档、部署说明和维护说明。
- `scripts/` 用于存放本地检查、数据处理或部署辅助脚本。
- `local-notes/` 用于存放本地草稿、参考资料和不需要上传 GitHub 的内容，必须保持在 `.gitignore` 中。

## 命名规范

- 目录和普通文件使用英文小写 kebab-case，例如 `daily-news`、`tech-trends`。
- 顶层约定文件可使用大写命名，例如 `README.md`、`AGENTS.md`、`LICENSE`。
- 页面模块名和导航路径保持一致，避免 README、页面链接和真实目录相互脱节。
- 新增目录前先确认是否会被 Git 追踪；空目录不会被 Git 保留，除非放置必要的占位文件。

## 内容范围约束

- **不转述任何单独高校的内部公告、通知、教务信息。** 这类信息具有强时效性和权威性，一旦我方更新不及时或存在遗漏，可能导致依赖的同学错过重要事项（如报名、考试、截止日期），进而产生责任风险。
- 涉及具体学校的内容，只收录该校面向社会公开发布、且具有普遍参考价值的信息（如公开招生简章、公开赛事通知），并在摘要中引导读者以官方渠道为准。
- 信息聚合以「行业趋势、技术动态、通用竞赛/考试政策」为主，不承担任何单一学校的消息送达职责。
- 所有自动化采集与编辑流程（包括 Hermes skill）必须遵守本约束，遇到高校内部公告一律丢弃。

## 静态站约束

- 页面应避免依赖服务器端渲染、数据库连接或私有运行环境。
- GitHub Pages 可能部署在仓库子路径下，资源链接优先使用相对路径，避免无必要的根路径引用。
- 页面导航指向的目标必须真实存在；未起步模块应先隐藏入口，或提供最小可用占位页。
- 公开仓库中的前端代码不得写入密钥、Token、私有 API 地址或个人敏感信息。

## 可维护性约定

- 共享样式放入 `assets/css/`，避免每个页面长期堆积大段内联样式。
- 共享交互逻辑放入 `assets/js/main.js`，页面专属逻辑放入对应页面脚本。
- 列表型内容应逐步从 HTML 中拆到 JSON 数据文件，由页面脚本渲染，降低后续自动采集和更新成本。
- 结构调整时同步更新 `README.md` 和相关文档。
- 本地草稿、Agent 生成包、临时文件和实验材料放入 `local-notes/` 或其他被忽略目录。

## 部署迁移原则

- GitHub Pages 阶段保持纯静态输出，确保直接打开或静态服务器预览都能访问。
- 迁移到云服务器时，优先将当前静态站作为前端根目录托管，不改变公开 URL 结构。
- 如果未来增加后端服务，前端通过配置化 API 基础地址访问后端，不把环境差异硬编码进页面。
- 数据采集、清洗和发布流程应和前端页面解耦，避免页面文件承担数据源职责。

## 变更前检查

- 新增或修改页面后，检查所有本地 `href` 和 `src` 是否能解析到真实文件。
- 修改目录结构后，检查导航、README 和部署文档是否同步。
- 发布前至少做一次本地静态预览，确认首页、栏目页、主题切换和筛选搜索等基础交互正常。
